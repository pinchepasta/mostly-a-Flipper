In `C:\Users\Eli\Downloads\Flipper`, several services/apps exist as **two copies** and only ONE is compiled. `tools/fam/generate.py`'s `load_manifests()` walks `MANIFEST_ROOTS = ["components", "applications", "applications_user"]` in priority order; on a duplicate `appid` the **first root wins**, so `components/**` beats `applications/**`. Confirmed live vs dead:
- Loader: **`components/loader/loader.c`** is live; `applications/services/loader/*` is dead.
- BT service: **`components/btshim/btshim.c`** is live (`appid="bt"`, TAG "BtSrv"); `applications/services/bt/bt_service/bt.c` is dead. The plan file for the driver/memory work wrongly names `applications/services/bt/bt_service/bt.c` for Phase 4b — the real target is `components/btshim/btshim.c`.
- BT settings UI: **`components/btshim/bt_settings_app.c`** is live; `applications/settings/bt_settings_app/*` is dead.

Verify which tree is live by grepping a unique log string from a boot capture (e.g. "BtSrv ready", "No PSRAM: deferring") rather than assuming. See [[flipper-cardputer-merge]] and [[flipper-esp-idf-build-env]].

**BLE / Flipper-mobile connection (fixed 2026-06-30):** btshim defers the BLE controller+Bluedroid+serial profile at boot on no-PSRAM boards (`#if !defined(CONFIG_SPIRAM)`, default BT off) to keep ~100 KB free for apps. `furi_hal_bt_start_radio_stack()` is a stub (`return true`); the real esp_bt_controller/bluedroid init is inside the serial profile `start()` (`components/ble_serial/ble_serial.c`, guarded by `serial_state.initialized`). The Settings toggle used to only call `furi_hal_bt_start_advertising()` (no profile → "no current profile!" → nothing advertised). Fix: `bt_handle_set_settings` now calls `bt_handle_start_stack`/`bt_handle_stop_stack` on the enabled transition. Verified: official Flipper mobile app discovers, pairs (`gap auth complete success=1`), runs RPC over BLE serial (UUIDs `19ed82ae-...228e6Xfe0000` in `components/ble_profile/extra_profiles/serial_profile.c`). Caveat: tight heap still throws `BLE_INIT: Malloc failed` on large TX packets (frame dropped, connection survives) — needs the memory-optimization phase. Note nano-printf (`CONFIG_NEWLIB_NANO_FORMAT=y`) does NOT support `%zu`, so those log lines print "zu" literally (drops the arg too — use `%u` + cast).

**BLE is fundamentally RAM-starved on this no-PSRAM board (confirmed 2026-07-01).** Total heap ~200 KB (heap_init: 149K+21K+32K DRAM+7K RTC). Desktop free **varies 62-88 KB** depending on which dolphin idle animation loaded (the animation is the swing factor). The BLE controller+Bluedroid stack is ~64 KB and an active RPC connection needs ~10 KB more (rpc_session_open ~6 KB + BtRxFeeder thread), so a full Flipper-mobile RPC session needs ~76-90 KB — right at/over the ceiling. It "worked" once by luck at peak free; normally it crashes OOM somewhere in the connect path (GATTS_CreateService, rpc_session_open stream buffer, or the feeder thread's xTaskCreate). **This is a hardware limit — cannot be fully fixed without PSRAM.** Memory work done (commit b0f1c6b): CONFIG_BT_CTRL_BLE_MAX_ACT 6->3 + CONFIG_BT_ACL_CONNECTIONS 4->2 in sdkconfig.defaults.m5stack_cardputer_adv freed ~17 KB and cured the enable-time crash; BT_RX_STREAM_SIZE 8192->2048 and BtRxFeeder 4096->2048 and GuiSrv stack 8192->4096 reclaimed more; a <12 KB free guard in bt_open_rpc_connection drops RPC (link stays connected-but-idle) instead of rebooting. **Key constraint: `tools/fam/generate.py` clamps every service stack to `ESP32_MIN_STACK_SIZE = 4096` (max(stack,4096)), so services can't go below 4096 — GuiSrv 8192->4096 was the only service-stack win available.** Measured per-task idle stack slack totaled ~43 KB but it's unreclaimable due to the clamp (kept for deep call chains). User de-prioritized BLE 2026-07-01 ("focus other than bluetooth").

**Measured heap ceiling / BLE Spam (2026-06-30):** with an app open, free internal heap is only **~46 KB** (desktop idle ~60-80 KB). The BLE controller + Bluedroid need ~64 KB, so heavy BLE apps cannot bring the stack up. BLE Spam (`applications/main/ble_spam`, incl. ble_walk) tears down the serial stack and re-inits its own Bluedroid + loads a 151-entry BLE UUID DB → OOM → ESP-IDF aborts (`BTU_StartUp`→`vQueueDelete(NULL)`, or controller `r_lld_core_init` interrupt-wdt). Three crash fixes landed (commits after 6959091): (1) `bt_alloc` calloc + explicit `bt->status=BtStatusOff` (BtStatusOff==1 not 0, else the stop guard never trips); (2) `ble_serial_reset_initialized` NULL-mutex guard; (3) heap guard in `ble_spam_hal_start`/`ble_walk_hal_start` (need ≥72 KB, else return false → scene backs out) AND `bt_handle_start_stack` early-returns when `!bt_settings.enabled` so app-exit `bt_start_stack` doesn't re-init a stack that was never up. Net: BLE Spam no longer crashes the device (returns to menu) but still can't RUN until Phase 4 frees RAM.
