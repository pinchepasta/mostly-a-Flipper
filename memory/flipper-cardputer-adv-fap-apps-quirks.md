**FAP loading is PSRAM-ONLY on this port — external `.fap`s CANNOT run on the no-PSRAM Cardputer-ADV.** `elf_file.c` forces `MALLOC_CAP_SPIRAM` for sections + `PSRAM_DATA_TO_INST` only maps PSRAM 0x3C→0x42. On the ADV `MALLOC_CAP_EXEC` free = **0**. I pushed the loader a long way (extended `PSRAM_DATA_TO_INST` to internal SRAM1 `0x3FC88000-0x3FD00000 → +0x6F0000`; `MALLOC_CAP_EXEC|8BIT` fallback for exec sections; **removed the debug "literal pool dump" in [flipper_application.c](components/flipper_application/flipper_application/flipper_application.c) `flipper_application_thread`** that data-read the fetch-only instruction alias and crashed pre-entry) — got it to actually CALL the FAP entry, but it still faults (EXCCAUSE 7, no exec heap). **Verdict: don't chase `.fap` on no-PSRAM; INTERNALIZE apps instead** (compile into firmware via `applications_user/`). buildFap.sh WAS taught the cardputer_adv target + needs Pillow in the idf py env; 3 symbols added to `firmware_api.c` (`round`, `__subdf3`, `sequence_display_backlight_on`) — left in place. User insists board has no PSRAM (a "Lumen PSRAM Fix" fw in their Downloads claims 8MB PSRAM but user says no; I tried `CONFIG_SPIRAM=y` OCT and reverted at their request — untested whether HW actually has it).
